1.2. Why Microservices?¶

Archimedes uses microservices to achieve its feature set. The reason this direction was selected was because of the inherent variations in the sort of setups one can expect in an environment usually found at a Mechanical Engineering company. It is not usual to find users on Linux machines, capable of POSIX commands such as qsub for instance.

Using microservices, one can entirely rely on HTTP commands to communicate with a job server. HTTP is so commonplace today, and it has vast acceptance for its ease of use.

A reason why Archimedes relies on microservices and not a monolith providing a REST API is because portions of the service cannot really exist on the same computer. Additionally, we have tried using queuing PBS and the associated posix commands, but there are limitations. One is that we have to rely on companies which develop these commands for extending the API or functionality, and we need to write wrappers around these commands for almost all functionality that we can gain through an in-house developed microservice framework.

Using the API calls, every single feature of the archimedes server can be accessed directly from any programming language or commandline of one’s choice. A user can submit a run through an Excel macro, from a powershell script, or a TCL/TK script from within a black-box software.

This is a great advantage of the microservice idealogy and this is why it was selected for this project. In addition, if the CAE world should ever accept mobile computing, Archimedes will be there first.

1.2.1. Service Discovery¶

Most of the services are hosted within docker containers. This makes things easy to keep track of. The way this is done is via a service-discovery design.

When a service launches, it is registered with a Service Discovery Agent, which is Consul in our case.

Service Discovery¶

The service discovery agent constantly checks if a service is healthy or otherwise, so all the services are designed with a /api/_health path so that they can respond regarding their health.

Additionally, since the containerized services are launched via docker, it is trivial to scale them and balance them across the swarm using docker’s inbuild features.

Some services can be launched multiple times to increase their performance, these include the SVN Linker, Project Manager and the Run Services.

1.2.2. Load Balancer¶

The load balancer is configured so that it can constantly point to the right URL for a service using the service discovery agent. This helps the services speak to each other without a complex logic between them. It also allows services to be launched dynamically.

Load Balancer¶

The load balancer we have used here is nginx but we could use alternates such as HAProxy in the future depending upon our satisfaction with Nginx.

1.2.3. Service Communication¶

When a user uses the Browser to send a request to Archimedes, the load balancer takes care of the redirection. This logic is kept hidden from the user. In fact, since the services may be scaled up or down at any instant, no one can know for certain where the services are running. This adds to the security and improves the way the entire system is run.

1.2.4. Message Queue¶

The Archimedes system is designed so that multiple run workers may consume the same information. To this end, a Message Queue is used in the backend. The message queue we chose is RabbitMQ. Whenever a user chooses an action that has some processing involved, such as searching for a file, or submission of a job to the cluster, an entry is made into a queue. This queue follows the Publisher/Subscriber model so that it can be consumed by multiple workers.

Multiple worker-based queues are optimum when there are simultaneous parallel tasks occurring in the pipeline. At times, the user doesn’t need to wait for these tasks to finish.

1.2.5. Planned Services¶

The services that are currently in the pipeline are as follows:

  1. Authentication service
    1. Allows clients to login.
    2. Provides authentication tokens (see JWT to understand how these work).
    3. Writes to a database with tokens.
    4. Records revoked tokens until their expiry.
  2. Application Database Service
    1. Serves a link to a database so everything is recorded and stored in the database.
    2. Provides backup mechanisms.
  3. Message Queue Service
    1. Provides a Message Queue in the Publisher/Subscriber model.
  4. Version Control Service
    1. Handles connections to the version control system.
    2. Handles hotlinking to files which cannot be stored in the VCS
    3. Moves files between regions/central server.
  5. Search Indexing Service
    1. Indexes key parameters/text/keywords from files which are queued in the search queue.
    2. Indexes older versions.
    3. Writes indexed data first into the database and then into the search provision service.
  6. Search Provision Service
    1. Consumes indexing data.
    2. Allows clients to search through the data.
  7. Project Management Service
    1. Allows creation of projects using different templates/inputs.
    2. Follows a poke-yoke system of creation.
  8. Task Management Service
    1. Queues tasks/jobs for execution.
    2. Provisions DAGs which are then queued on prospective machines using the run services.
  9. Run Services
    1. Consumes DAGs written into the queue.
    2. Executes each operator of a DAG in one of the free-corresponding machines.
